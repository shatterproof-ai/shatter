# Unified Plan: Cached Harness Compilation for Compiled-Language Frontends

Synthesized from three sources:

- Original plan (`replicated-noodling-axolotl.md`): identified the core problem and proposed stdin-based inputs
- Critique A (`harness-compilation-critique.md`): feature-gated in-crate compilation, XDG cache, persistent subprocess, TS architecture lessons
- Critique B (`2026-03-21-replicated-noodling-axolotl-review.md`): semantic fidelity as primary constraint, Go package-level compilation, comprehensive cache keys, versioned cache with atomic writes

## Problem Statement

The Rust frontend compiles a fresh Cargo project for every execute call: instrument source, generate harness with inputs baked in, `cargo build`, delete everything. ~35s per call. The Go frontend has the same structural issue (single-file copy, not package-level), masked by faster compile times.

Both frontends violate a property the TypeScript frontend gets right: **separate the compiled artifact from the execution invocation, so inputs never trigger recompilation.**

## Design Principles

These are ordered. When principles conflict, higher-ranked ones win.

1. **Semantic fidelity**: code must compile and run with the same meaning as in the user's real crate/module. No import rewriting. No simulating crate structure across a dependency boundary.
2. **Compile once, execute many**: inputs passed at runtime (stdin), never baked into source.
3. **Minimal in-tree footprint**: one stable generated file per language, feature-gated to be invisible to normal builds. Everything else external.
4. **Let the build tool handle build staleness**: Shatter tracks instrumentation staleness only. Cargo/Go handle compilation staleness.
5. **Persistent execution**: warm iterations should approach TS-like speed (<5ms) via long-lived subprocess.
6. **Graceful degradation**: build failures fall back to the legacy approach. Users are never worse off.

## Architecture

### Rust: Feature-Gated Single Dispatch Binary (`bin_only` mode — default)

Generate a single dispatch binary inside the user's crate, feature-gated so it has zero impact on normal builds. This is the `bin_only` harness mode, which is the default. For crate-private access, see the `crate_bridge` opt-in mode described below.

#### One-time Cargo.toml addition

Done by `shatter init` or automatically on first execute with a log message:

```toml
# shatter-autogenerated
[features]
shatter-harness = ["dep:serde_json"]

[dependencies]
serde_json = { version = "1", optional = true }

[[bin]]
name = "shatter-harness"
path = "src/bin/__shatter_exec.rs"
required-features = ["shatter-harness"]
```

Properties:

- **O(1)**: one feature, one binary, one dependency. Never grows regardless of how many functions are explored.
- **Invisible**: `cargo build` without `--features shatter-harness` skips it entirely. `cargo test` ignores it. `cargo clippy` ignores it.
- **No workspace conflict**: no separate `Cargo.toml`. The `[[bin]]` is part of the user's existing crate manifest.

Crash-safe management:

1. Before modifying, write a `.pending` sentinel in XDG cache with original `Cargo.toml` SHA-256.
2. Append via atomic write (write full file to `.tmp`, rename).
3. On startup, if `.pending` exists, verify and clean.
4. The additions are permanently inert without the feature flag, so leaving them is safe. `shatter clean` removes them.

#### Generated dispatch file

`src/bin/__shatter_exec.rs` (gitignored):

```rust
// shatter-autogenerated -- safe to delete
use std::io::{self, BufRead, Write};

fn main() {
    let stdin = io::stdin();
    let stdout = io::stdout();
    for line in stdin.lock().lines() {
        let line = match line {
            Ok(l) => l,
            Err(_) => break,
        };
        let req: serde_json::Value = match serde_json::from_str(&line) {
            Ok(v) => v,
            Err(e) => {
                let err = serde_json::json!({"error": e.to_string()});
                serde_json::to_writer(&mut stdout.lock(), &err).unwrap();
                println!();
                continue;
            }
        };
        let func = req["function"].as_str().unwrap_or("");
        let result = dispatch(func, &req);
        let mut out = stdout.lock();
        serde_json::to_writer(&mut out, &result).unwrap();
        out.write_all(b"\n").unwrap();
        out.flush().unwrap();
    }
}

fn dispatch(func: &str, req: &serde_json::Value) -> serde_json::Value {
    // Generated match arms, one per explored function.
    // Each arm: deserialize inputs, reset runtime, catch_unwind(|| call),
    // serialize result + branch_path + side_effects.
    match func {
        "classify_number" => { /* generated */ }
        "auth::validate::check" => { /* generated */ }
        _ => serde_json::json!({"error": format!("unknown function: {func}")})
    }
}
```

Key properties:

- **Public API access**: the `[[bin]]` target is a separate crate that depends on the library. It can call any `pub` function and use any `pub` type. It **cannot** name `pub(crate)` types in function signatures (parameters or return types). Functions that internally use `pub(crate)` items work fine — the visibility boundary only affects what the harness source must name at compile time. This limitation is inherent to Rust's crate model; no generated file arrangement avoids it without modifying user source code. Functions with `pub(crate)` signature types fail with a clear compiler error and fall back to the legacy path.
- **No import rewriting**: the harness references the library by crate name (e.g., `my_crate::function`). No `crate::` rewriting needed.
- **Feature flag inheritance**: gets whatever features the crate has active, plus `shatter-harness`.
- **Macro correctness**: the library compiles in its real crate context; macros expand normally. The binary links against the compiled library.
- **Workspace correctness**: the `[[bin]]` is in a workspace member, resolved normally.

Dispatch table growth: O(n) in one file. Cargo's incremental compilation means adding a new function arm recompiles only `entry.rs` (~2-3s incremental + relink).

#### Persistent subprocess

The `for line in stdin` loop keeps the binary running. The frontend handler maintains one running subprocess per project:

- **Warm iteration (same function, different inputs)**: send JSON line to stdin, read JSON line from stdout. <5ms.
- **New function**: regenerate `entry.rs` with new dispatch arm, `cargo build`, restart subprocess. ~3s.
- **Source change**: re-instrument, regenerate affected dispatch arm, `cargo build`, restart. ~3s.
- **Process crash**: detected by broken pipe or EOF. Restart transparently.

#### `no_std` crates

The harness needs `serde_json` and `std::io`. If the user's crate is `no_std`, the feature gate also enables `std`:

```toml
shatter-harness = ["dep:serde_json", "std"]
```

This requires detecting `no_std` (look for `#![no_std]` in `lib.rs`) and adjusting the feature definition. If the crate truly cannot link `std` (embedded target), fall back to the standalone approach.

#### `crate_bridge` mode (opt-in alternative)

The `bin_only` mode above cannot access `pub(crate)` types in function signatures
because the `[[bin]]` target is a separate crate. For functions that require
crate-private access, the `crate_bridge` mode injects a feature-gated wrapper
module directly into the library crate:

- Adds a `#[cfg(feature = "shatter-harness")]` module to `lib.rs`
- The wrapper module can call `pub(crate)` functions and name `pub(crate)` types
- A separate driver binary depends on the library with the feature enabled
- Build artifacts go to a stable directory in XDG cache

This mode is opt-in via the `harness_mode` field on execute requests. It requires
modifying the user's `Cargo.toml` and `lib.rs`, but the additions are clearly
marked as autogenerated and gated behind the `shatter-harness` feature.

### Go: Package-Level Compilation with Dispatch Harness

The current Go frontend copies a single file to a temp directory, losing package-private helpers, `init()` functions in sibling files, and build-tag semantics. The revised design compiles at package granularity.

#### Build in the real module context

Instead of copying one file to a temp dir:

1. Copy ALL `.go` files from the target package directory to the build directory.
2. Instrument the target file (add branch recording to the target function).
3. Leave sibling files unmodified -- they provide helpers, types, and `init()` functions.
4. Rewrite all files to `package main`.
5. Add `shatter_recorder.go` (branch tracking functions) and `main.go` (dispatch harness).
6. Copy `go.mod` and `go.sum` from the module root, making relative `replace` directives absolute.

This preserves package semantics: unexported helpers compile, `init()` functions run, build tags are preserved (the full set of files from the package directory is present).

#### Build directory

Go build output goes to XDG cache (see Storage section):

```
$XDG_CACHE_HOME/shatter/{project-hash}/go/
  go.mod
  go.sum
  *.go                    # all package files + instrumented target + recorder + harness
  shatter-harness         # compiled binary
```

All files are `package main` in one flat directory. No `cmd/`+`modules/` split -- that introduces package boundary problems the current flat approach doesn't have.

#### Dispatch harness

Same pattern as Rust: one `main.go` with a function-name dispatcher, persistent subprocess, stdin/stdout JSON protocol. The dispatcher identifies functions by name from the request payload.

#### Go workspaces

If `go.work` exists at the module root, copy it and adjust paths. If the module is part of a multi-module workspace, the `replace` directives must account for sibling modules.

### Standalone Files

Files without a parent `Cargo.toml` or `go.mod`:

- Create a minimal project in `$XDG_CACHE_HOME/shatter/standalone-{lang}/`
- Wrap source in module (Rust) or generate minimal `go.mod` (Go)
- No import rewriting needed (standalone files have no intra-crate imports)
- Stdin-based inputs, debug builds
- Falls back to single-shot execution (no persistent subprocess, since the project isn't stable)

## Storage

Four lifecycle-separated locations, following str-y3po:

| Category | Location | Lifecycle | Contents |
|---|---|---|---|
| In-tree harness | `src/bin/__shatter_exec.rs` (gitignored) | Stable, session-spanning | Dispatch harness, instrumentation hashes in XDG cache |
| In-tree manifest | `Cargo.toml` additions | Permanent (inert without feature flag) | `[[bin]]` + `[features]` |
| Build cache | `$XDG_CACHE_HOME/shatter/{project-hash}/` | Evictable, survives reboot | Go build dir, compiled binaries, cache metadata |
| Preserved artifacts | `shatter-artifacts/` | User-owned, durable | Behavior maps, traces, exports |
| Temp | System temp dir | Per-run, disposable | Scratch files, transient logs |

For Rust, compiled artifacts live in the crate's normal `target/` directory (the `[[bin]]` target builds there naturally). No separate cache directory needed.

For Go, the build directory is external (XDG cache) because Go doesn't have an equivalent to Cargo's feature-gated in-crate binary.

### XDG Cache Directory

`$XDG_CACHE_HOME` defaults to `~/.cache` on Linux, `~/Library/Caches` on macOS, `%LOCALAPPDATA%` on Windows. Use Rust `dirs::cache_dir()` for cross-platform resolution.

```
~/.cache/shatter/
  {sha256-of-project-root}/
    go/
      go.mod, go.sum, *.go, shatter-harness, source-hashes.json
    cache-meta.json         # cache key metadata (see Staleness section)
  standalone-rust/
    Cargo.toml, src/, target/
  standalone-go/
    go.mod, *.go
```

### Cleanup

- **`shutdown` protocol command**: kill persistent subprocess. Leave cache intact.
- **`shatter clean` CLI command**: remove XDG cache for the project, remove `src/bin/__shatter_exec.rs`, optionally remove Cargo.toml additions.
- **Orphan detection**: on startup, scan dispatch table arms against existing source files. Remove stale arms (one `stat` per function).
- **Disk budget**: log cache size on startup if >1GB. Future: `max-cache-size` config with LRU eviction.

## Staleness

### Two-layer strategy

| Layer | What changes | Who detects it | Action |
|---|---|---|---|
| Instrumentation | Source file content | Shatter (SHA-256 of source) | Re-instrument, regenerate dispatch arm |
| Compilation | Everything else | Cargo / `go build` | Recompile (or no-op if unchanged) |

Shatter hashes source files to decide whether to **re-instrument**. It does NOT decide whether to **recompile** -- that is delegated to the build tool.

### Instrumentation cache key

The decision to re-instrument should include more than just the source file hash. Full key:

| Component | Why |
|---|---|
| Source file SHA-256 | Content changed |
| Shatter instrumentor version | Instrumentation logic changed |
| Shatter runtime version | Runtime API changed |
| Build mode (debug/release) | Affects codegen |
| Feature flags (Rust) | Affects conditional compilation |

If any component changes, re-instrument and regenerate the dispatch arm. Then run `cargo build` / `go build`, which will recompile if the regenerated source differs from what's cached.

Manifest/lockfile/toolchain changes are NOT in Shatter's cache key -- those are Cargo/Go's responsibility. If `Cargo.lock` changes, `cargo build` will recompile. If the Go toolchain updates, `go build` will recompile. Shatter doesn't need to detect these because it always runs the build tool (which handles them correctly).

### Atomic writes and versioning

- Generated files (`src/bin/__shatter_exec.rs`, `source-hashes.json`) are written via temp-file-plus-rename (atomic on POSIX).
- Cache directory includes a `cache-meta.json` with a layout version number. If the version doesn't match the running Shatter version, the entire cache directory is invalidated and rebuilt.
- If a compiled binary is missing but hashes say "fresh" (manual deletion, `cargo clean`), detect by checking binary existence before executing. If missing, rebuild.

## Fallback

If the in-crate harness build fails (feature incompatibility, `no_std` without `std` available, unexpected crate structure):

1. Log the full error: `[shatter-rust] in-crate harness build failed: {error}. Falling back to direct compilation.`
2. Attempt the legacy direct-compilation path (current approach: temp project, wrap source in module, single-shot execution).
3. If the fallback also fails, surface both errors with context.

Users are never worse off than the current approach. The fallback is the escape hatch for edge cases we haven't anticipated.

## What TypeScript Gets Right (Lessons for All Frontends)

The TS frontend achieves <1ms warm iterations through three properties:

| Property | TS implementation | Compiled-language equivalent |
|---|---|---|
| In-memory compilation | `ts.transpileModule()` + cached `vm.Script` | Feature-gated `[[bin]]` with incremental Cargo |
| Artifact reuse | `script.runInContext(sandbox)` with new inputs | Persistent subprocess with stdin/stdout JSON |
| Shared process | `vm.createContext()` isolation | Single dispatch binary handling all functions |

### Future frontend guidance

| Language | Architecture | Rationale |
|---|---|---|
| Python | TS-like (in-process) | `compile()` + `exec()` with restricted globals. Zero filesystem overhead. |
| Ruby | TS-like (in-process) | `RubyVM::InstructionSequence.compile`. Less isolated than TS. |
| Java | TS-like (in-process) | `javax.tools.JavaCompiler` + `ClassLoader` isolation. |
| Lua | TS-like (in-process) | Native `load()` + separate `_ENV` tables. Ideal for embedding. |
| C/C++ | Persistent subprocess | No safe in-process isolation. TCC offers partial in-memory compilation. |

Interpreted/JIT languages should follow the TS pattern. Compiled languages should follow the persistent-subprocess pattern described in this plan.

## Implementation Phases

### Phase 1: Stdin-Based Inputs (P1, highest value, lowest risk)

**Issue**: str-9u7n

- Rust: change `generate_harness()` to read inputs from `serde_json::from_reader(stdin())`. Change `execute_function_with_timing()` to pipe inputs to subprocess stdin.
- Go: change `generateHarness()` to read from `json.NewDecoder(os.Stdin)`. Change `ExecuteFunctionWithTiming()` to pipe inputs via stdin.
- No architectural changes. Just modify the harness template and subprocess invocation.

**Stop gate**: measure walkthrough step 10 timing after this phase. If it passes the 120s budget, remaining phases are optimization. Proceed anyway for the semantic fidelity and speed goals, but with reduced urgency.

### Phase 2: Debug Builds + Warm Cache Preservation

**Issue**: str-tc2h

- Default to `cargo build` (debug) instead of `cargo build --release`. 2-5x faster compilation.
- Stop deleting temp directories after each execute. Reuse the build cache across iterations within a session.
- Offer `--release` or `SHATTER_HARNESS_RELEASE=1` for performance-sensitive targets.

### Phase 3: In-Crate Harness (Rust) + Package-Level Build (Go)

- Rust: implement the feature-gated `[[bin]]` approach. Generate `src/bin/__shatter_exec.rs` with dispatch table. Build via `cargo build --bin shatter-harness --features shatter-harness`.
- Go: copy all package files (not just the target file) to the build directory. Instrument the target file only. Rewrite all to `package main`. Build in XDG cache directory with absolute `replace` directives.
- Implement instrumentation cache key (source hash + instrumentor version + runtime version + build mode + features).
- Implement atomic writes for generated files and cache metadata.
- Implement fallback to legacy approach on build failure.

### Phase 4: Persistent Subprocess (deferred)

**Issue**: str-lefp
**Status**: not yet implemented. Deferred until profiling data from phases 1-3
justifies the added complexity. The current architecture uses single-shot
harness execution per request.

- Change the dispatch binary from single-shot to a stdin/stdout REPL loop.
- Frontend handler maintains one running subprocess per project.
- On new function or source change: regenerate, rebuild, restart.
- On crash: detect broken pipe, restart transparently.
- Warm iteration cost: <5ms.

### Phase 5: Parallel Compilation + Precompiled Template

**Issues**: str-0pp0, str-o650, str-jalx

- Compile multiple harness targets in parallel for multi-function cold starts.
- Factor common harness boilerplate into a precompiled library to shrink per-arm compilation units.
- Use `cargo check` for validation-only paths (fast feedback without full build).
- Drive by profiling data, not speculation.

## Open Questions

1. **Cargo.toml addition: permanent or opt-in?** The `required-features` gate makes it inert, but some users may object to any modification. Options: (a) add automatically on first execute with a log message, (b) require `shatter init` as an explicit step. Recommendation: auto-add with clear logging and `shatter clean` to undo.

2. **Multi-crate workspaces**: each crate with explored functions needs its own `[[bin]]` + feature. Still O(1) per crate, but O(crates) total. Is that acceptable, or should there be a single workspace-level harness?

3. **Dispatch table size at scale**: at hundreds of functions, the dispatch table may slow incremental compilation. Mitigation: split by source file (one dispatch function per file, called from the main dispatcher). This keeps each compilation unit small while maintaining O(1) binary count.

4. **Go `cmd/` naming**: the dispatch harness lives in the XDG cache dir, not in the user's `cmd/` tree. This is intentional -- `cmd/` entries in the user's project are their own code, not ours.

5. **Concurrent sessions**: deferred. Document that concurrent Shatter sessions against the same project are unsupported in v1. The persistent subprocess model naturally serializes within a session (protocol is request-response). Cross-session conflicts would require file locking with stale-lock recovery -- not worth the complexity until users hit it.
