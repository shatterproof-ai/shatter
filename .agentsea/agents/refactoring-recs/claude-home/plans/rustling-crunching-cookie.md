# Plan: str-3ky9.1 — MockParam type and build_mock_params

## Context

The auto-mock system needs a `MockParam` struct that parallels `ParamInfo` but for mock return values. This bridges the gap between `ExternalDependency` (what we detected) + `MockConfig` (how to mock it) and a unified struct the orchestrator can use to set up mocks.

## File to modify

- `shatter-core/src/auto_mock.rs`

## Implementation

### 1. Add `ValueSource` enum

Describes where the mock's return value comes from:
- `AutoGenerated` — generated by classify+default logic
- `UserOverride` — from `.shatter/config.yaml`
- `BehaviorMap` — from an existing behavior-map mock

### 2. Add `MockParam` struct

```rust
pub struct MockParam {
    pub symbol: String,
    pub return_type: TypeInfo,
    pub category: IoCategory,
    pub call_count_estimate: u32,
    pub value_source: ValueSource,
}
```

Derive: `Debug, Clone, PartialEq`

### 3. Add `build_mock_params()` function

Signature:
```rust
pub fn build_mock_params(
    deps: &[ExternalDependency],
    configs: &[MockConfig],
) -> Vec<MockParam>
```

Logic:
- Build a HashMap of configs keyed by symbol
- For each dep, classify it via `classify_dependency`
- Skip if category is `PureUtility` and no config exists (passthrough)
- Look up matching config; determine `ValueSource` based on whether config exists
- `call_count_estimate` = `dep.call_sites.len() as u32`
- Collect into `Vec<MockParam>`

### 4. Unit tests

- Empty deps → empty result
- All passthrough (pure utility) deps → empty result
- Mixed deps with configs → correct MockParams with right categories and value sources
- Dep with no matching config → still produces MockParam with AutoGenerated source
- Dep with matching config → UserOverride source

### 5. Proptest

`MockParam` doesn't serialize and has no complex invariants beyond what unit tests cover. Skip proptest — specific examples are clearer here per the policy.
