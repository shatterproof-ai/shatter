## 3. Findings

### P1 — Rust harness duplicates `panic_impl` for std builds

**Evidence:** Running the Rust fixture after building `shatter-rust` failed both attempted functions. `contains` failed with `error[E0152]: found duplicate lang item panic_impl`; the generated harness inserted a `#[panic_handler]` while compiling a `std` binary. `zolem_match` failed with `bin_only harness incompatible` for a raw pointer parameter and suggested crate bridge mode.

**Impact:** A user who follows the hint to build `shatter-rust` still cannot explore this realistic Rust fixture. This makes Rust support look present in discovery but broken at execution time.

**Action:** Fix Rust harness generation so `#[panic_handler]` is only emitted for `no_std` contexts, and route raw-pointer or extern-style functions to a clear unsupported classification unless crate bridge execution is implemented and selected.

### P1 — Go single-target `explore --project-dir <nested module>` writes into audited repos

**Evidence:** A targeted explore of `api/internal/search/access.go:IsFieldSearchable` with `--project-dir /home/ketan/project/kapow/api`, `--no-cache`, `--no-seeds`, and external `--output` auto-created `/home/ketan/project/kapow/api/.shatter/` and wrote `/home/ketan/project/kapow/api/shatter-artifacts/explore-results/...`. I removed those generated files after the run. The scan path already has an external-audit guard in `shatter-cli/src/main.rs` that skips implicit init when `-o`, `--no-cache`, and `--no-seeds` are combined; `explore` lacks the same protection.

**Impact:** External audit runs can dirty the target repository, including nested modules, even when the user clearly requested external output and disabled caches/seeds.

**Action:** Apply the scan command's external-audit mode to `explore`: when all outputs are external and both cache/seeds are disabled, skip implicit init and suppress project-local explore artifacts unless explicitly requested.

### P2 — Go scan timeout classification hides setup/build cost and actionable cause

**Evidence:** The bounded Go sample selected 9 functions in `api/internal/search/*.go`; 3 completed and 6 failed with only `timed out after 8s`. A later targeted run of `IsFieldSearchable` against the nested module completed in 17.1s with `--timeout-explore 30`, implying the 8s failure was likely setup/build/execution overhead rather than a genuinely hanging function.

**Impact:** The report tells the user which functions timed out, but not whether the timeout was analysis, setup, build, first execution, or exploration. This makes it hard to decide whether to raise `--timeout-per-fn`, adjust `--build-timeout`, reduce parallelism, or file a frontend bug.

**Action:** Split timeout reporting by phase in scan results and progress events. At minimum, record elapsed build/setup/execute/explore time and include the phase in `failed[].reason`.

### P2 — `--timeout-total` did not promptly bound full dry-run analysis

**Evidence:** `shatter scan /home/ketan/project/kapow --dry-run --no-cache --no-seeds --output ... --timeout-total 120` was still batch-analyzing after nearly two minutes and had to be interrupted. It had discovered 871 files, skipped Rust due to missing frontend, then continued through batch analysis.

**Impact:** Dry-run is the command users reach for before a broad scan. On a large repo, a dry run that is not promptly bounded by `--timeout-total` is risky in CI and frustrating interactively.

**Action:** Enforce the total scan deadline during dry-run discovery and batch analysis, not only during exploration. Return a partial dry-run plan with a clear timeout status.

### P2 — Rust frontend distribution hint is incomplete for workspace users

**Evidence:** Initial broad discovery warned: `shatter-rust frontend not found: install it on PATH or build with cargo build --manifest-path shatter-rust/Cargo.toml`. Following that command built the binary at `shatter-rust/target/release/shatter-rust`, not the workspace `target/release`. Shatter still failed until that nested target directory was explicitly prepended to `PATH`.

**Impact:** The hint is technically true but incomplete. A user can run the suggested build and still get the same failure if they expect the frontend to be where the CLI release binary lives.

**Action:** Improve the hint to say where the binary lands and how to run with it, or make the CLI discover the sibling `shatter-rust/target/release/shatter-rust` when running from a source checkout.

### P2 — `scan --seed` only accepts integers, but help does not say so

**Evidence:** `--seed kapow-audit` failed with `invalid digit found in string`. The help text says "Seed for deterministic core sample selection" but does not specify numeric-only input.

**Impact:** Users naturally try mnemonic seeds for reproducible audits. The failure is easy to recover from, but it is unnecessary friction.

**Action:** Either accept arbitrary strings by hashing them to `u64`, or update help and error text to say the seed must be an unsigned integer.

### P2 — JSON output behavior differs across `scan` and `explore`

**Evidence:** `scan --dry-run --format json --stdout` failed because dry-run stdout is markdown-only, while `explore --format json --stdout` failed with a different message: "JSON output for explore writes spec bundle; use `-o <file>.json`..." In a subsequent failed `explore -o file.json`, Shatter exited 0 after an analyze error and wrote no file.

**Impact:** Agents and CI wrappers cannot rely on consistent JSON/stdout semantics or exit/file guarantees across adjacent commands.

**Action:** Define one JSON contract for CLI commands: either JSON stdout is supported consistently, or unsupported combinations fail before work starts with a uniform diagnostic. If `-o file.json` is accepted, a failure should either write a structured failure document or exit nonzero before promising output.

### P3 — TypeScript scans include low-value hook/re-export targets by default

**Evidence:** The TypeScript sample completed 8/8 quickly, but included React hook wrappers and generated-client wrappers such as `usePostHogIdentify`, `useStudentProfile`, and `useQuery`. These completed with low behavioral value: e.g. `usePostHogIdentify` mocked `useEffect` and `useRef`, covered 0/2 branches, and produced one `null` return behavior.

**Impact:** The run succeeds, but the default target set spends budget on framework glue that is less useful than pure domain helpers.

**Action:** Add or document scope presets for frontend projects that prefer pure functions and domain modules over React hooks, generated GraphQL wrappers, and component glue.

Type: `bug`

Title: `Rust harness panic_impl duplicate`

Description:

### Explore external audit mode

Type: `bug`

Title: `Explore external audit writes target repo`

Description:
`shatter explore <target> --project-dir <nested-module> --no-cache --no-seeds -o <external-file>` auto-initializes `.shatter/` and writes `shatter-artifacts/` inside the audited project. `scan` already guards this external-audit mode in `shatter-cli/src/main.rs`; `explore` should match it.

Acceptance:
- External-output explore runs with `--no-cache --no-seeds` do not create `.shatter/`, `.shatter-cache/`, or `shatter-artifacts/` in the target repo.
- Existing artifact-writing behavior remains available when the user explicitly requests project-local artifacts.
- Regression test covers a nested Go module with external output.

### Scan dry-run timeout deadline

Type: `bug`

Title: `Scan dry-run ignores total timeout`

Description:

Acceptance:

- On deadline, dry-run exits with a clear timeout status and partial plan summary.
