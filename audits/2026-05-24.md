## 6. Suggested Issues

### Filed P1: `str-jssd` Go search scan all timeouts

Type: `bug`

Problem: The Kapow Go search scan sample regressed from 3/9 completed on 2026-05-21 to 0/9 completed on 2026-05-24 using the same include pattern, seed, sample size, iteration cap, and timeout settings.

Acceptance:

- Reproduction scan completes at least the previously successful functions, or classifies failures with deterministic unsupported/setup/build reasons.
- Test or fixture captures the regression class: exported Go methods/functions in a nested module should not all time out.
- Failure report distinguishes actual execute timeout from setup/build/request timeout.

### Filed P1: `str-hbc6` External Go scan writes launchers

Type: `bug`

Problem: `scan`/Go execution with external output plus `--no-cache --no-seeds` still creates `api/.shatter-launchers/` in Kapow.

Acceptance:

- External audit scan does not leave `.shatter-launchers/` or other untracked project-local generated launcher artifacts.
- Regression test uses a fixture git repo and checks `git status --short` remains clean.

### Filed P2: `str-84vt` Rust fixture still times out

Type: `bug`

Problem: The Rust `match.rs::contains` fixture no longer fails with duplicate `panic_impl`, but now times out with `timed out after 8s` after roughly 38s elapsed.

Acceptance:

- `contains` either completes or reports a deterministic setup/build/unsupported reason.
- Timeout report phase and elapsed time are consistent.

### Filed P2: `str-q3bm` TS error-only summary count

Type: `feature`

Problem: Framework-wrapper functions such as `useQuery` can complete as `error_only` with React-context `TypeError` and still count in the completed total. The filed issue is intentionally scoped to adding a separate machine-readable `error_only` summary count while preserving legacy `completed` semantics; broader scan presets/docs for excluding hooks are out of scope.

Acceptance:

- Scan JSON summary includes a separate machine-readable count for `completion_outcome: "error_only"` results.
- Existing `completed` count remains backward-compatible unless the schema is explicitly versioned and documented.
- A focused fixture or test proves an `error_only` TypeScript result is counted separately from normal completed behavior.
